# Configuration — multi-file env layout

This is the operator-facing reference for the per-concern env-file layout introduced in macf#342 (v0.2.18+). It replaces the pre-#342 monolithic `claude.sh` (which inlined all env exports in one file) with seven dedicated files under `<workspace>/.claude/.macf/env.*`, each owned by a single concern.

If you operated MACF on v0.2.17 or earlier, your launcher inlined identity / GitHub-auth / cert / registry / telemetry / tmux exports directly in `claude.sh`. After upgrading to v0.2.18+, `macf update` auto-detects the monolithic shape and migrates it in place — see [Rollback](#rollback) if you need to undo.

## Overview

The post-#342 layout:

```
<workspace>/
├── claude.sh                          # thin launcher (sources env files + tmux self-wrap; no inlined env exports)
├── .claude/.macf/                     # per-concern env files (sourced alphabetically by claude.sh on launch)
│   ├── env._helpers                   # library: macf_settings_get jq helper — sourced FIRST (underscore prefix)
│   ├── env.certs                      # MACF_CA_CERT/KEY, MACF_AGENT_CERT/KEY, MACF_LOG_PATH
│   ├── env.github                     # APP_ID, INSTALL_ID, KEY_PATH, GH_TOKEN mint, GIT_AUTHOR_NAME / COMMITTER_NAME
│   ├── env.identity                   # MACF_PROJECT, MACF_AGENT_NAME, MACF_AGENT_ROLE, MACF_AGENT_TYPE, MACF_WORKSPACE_DIR
│   ├── env.registry                   # MACF_REGISTRY_TYPE + per-type vars (MACF_REGISTRY_REPO / ORG / USER / PATH)
│   ├── env.telemetry                  # 3 OTel mandatory gates + per-signal exporters + 4-layer endpoint chain
│   └── env.tmux                       # MACF_TMUX_SESSION, MACF_TMUX_WINDOW (for the on-notify wake path)
└── .macf/                             # channel-server runtime (certs, logs, plugin) — separate concern
```

`claude.sh` becomes a thin orchestration script. Its body, after `set -euo pipefail` + `SCRIPT_DIR` resolution, is essentially:

```bash
if [ -d "$SCRIPT_DIR/.claude/.macf" ]; then
  for f in "$SCRIPT_DIR/.claude/.macf"/env.*; do
    [ -f "$f" ] && source "$f"
  done
fi
```

Shell glob expansion sorts alphabetically; the underscore-prefixed `env._helpers` sorts BEFORE alphabetical-letter siblings (`env.certs`, `env.github`, etc.), so its function definitions (`macf_settings_get`) are available when `env.identity` and `env.telemetry` source later and call them.

## macf-managed vs operator-managed boundary

The seven files split into three classes by ownership and refresh semantics:

| File | Owner | `macf update` behavior |
|---|---|---|
| `env._helpers` | macf-managed (library) | Overwrite + warn-once on hand-edit |
| `env.identity` | macf-managed | Overwrite + warn-once on hand-edit |
| `env.github` | macf-managed (empty in local-mode per [DR-024](../design/decisions/DR-024-local-registry-mode.md)) | Overwrite + warn-once on hand-edit |
| `env.certs` | macf-managed | Overwrite + warn-once on hand-edit |
| `env.registry` | macf-managed | Overwrite + warn-once on hand-edit |
| `env.telemetry` | **operator-managed** | Bootstrap-write if absent; preserve unconditionally otherwise |
| `env.tmux` | **operator-managed** | Bootstrap-write if absent; preserve unconditionally otherwise |

**macf-managed**: regenerated by `macf update` from the canonical templates in `packages/macf/src/cli/env-files.ts`. Hand-edits are detected (content drift vs fresh generator output), warned about on stderr, and overwritten. For local customization, use the `env.local.<name>` operator-override convention (see [Operator-custom convention](#operator-custom-convention) below) or `.claude/settings.local.json` `env` block (read at runtime via `macf_settings_get`).

**operator-managed**: bootstrap-written on first `macf update` if absent (so the workspace starts with sensible defaults), then preserved unconditionally. Operator owns the contents; `macf update` will not touch them once they exist.

## File-by-file reference

### `env._helpers` — library (macf-managed)

**Owner**: macf-managed library file. The underscore prefix sorts BEFORE alphabetical-letter siblings, so functions defined here are available when downstream env files source.

**Vars defined**: none. This file defines shell **functions**, not variables — sourcing it has no observable effect beyond making functions callable.

**Functions exported**:

- `macf_settings_get(var_name)` — reads `.env.<var_name>` from `<workspace>/.claude/settings.local.json` via `jq`. Returns empty string if the file/key is missing or `jq` isn't installed. Used by `env.identity` (3-layer settings priority for `MACF_AGENT_NAME` / `MACF_AGENT_ROLE`) and `env.telemetry` (4-layer endpoint chain for `MACF_OTEL_ENDPOINT`).

**Common operator edits**: do not edit. This is a library file — overrides belong in `env.local.<name>` or `.claude/settings.local.json`.

**Example content**:

```bash
macf_settings_get() {
  local var_name="$1"
  if [ -f "$SCRIPT_DIR/.claude/settings.local.json" ] && command -v jq >/dev/null 2>&1; then
    jq -r ".env.${var_name} // empty" "$SCRIPT_DIR/.claude/settings.local.json" 2>/dev/null
  fi
}
```

### `env.identity` — macf-managed

**Owner**: macf-managed.

**Vars defined**: `MACF_WORKSPACE_DIR`, `MACF_PROJECT`, `MACF_AGENT_TYPE`, `MACF_AGENT_NAME`, `MACF_AGENT_ROLE`.

`MACF_AGENT_NAME` and `MACF_AGENT_ROLE` preserve the post-macf#313 settings-driven 3-layer priority (env > `.claude/settings.local.json` `env` block > baked default). The settings-layer read uses `macf_settings_get` from `env._helpers` (sourced first by alphabetical glob order).

**Common operator edits**: do not edit. To override identity vars, use one of:

- **One-shot**: prefix the launcher invocation: `MACF_AGENT_NAME=foo ./claude.sh`
- **Persistent across runs**: `.claude/settings.local.json` `env.MACF_AGENT_NAME`
- **Persistent + file-based**: create `env.local.identity` with your override (see [Operator-custom convention](#operator-custom-convention) below)

**Example content** (snippet):

```bash
export MACF_WORKSPACE_DIR="$SCRIPT_DIR"
export MACF_PROJECT="my-project"
export MACF_AGENT_TYPE="permanent"

MACF_AGENT_NAME="${MACF_AGENT_NAME:-$(macf_settings_get MACF_AGENT_NAME)}"
MACF_AGENT_NAME="${MACF_AGENT_NAME:-code-agent}"
export MACF_AGENT_NAME

MACF_AGENT_ROLE="${MACF_AGENT_ROLE:-$(macf_settings_get MACF_AGENT_ROLE)}"
MACF_AGENT_ROLE="${MACF_AGENT_ROLE:-implementer}"
export MACF_AGENT_ROLE
```

### `env.github` — macf-managed (empty placeholder in local-mode)

**Owner**: macf-managed.

**Vars defined** (GitHub-mode workspaces): `APP_ID`, `INSTALL_ID`, `KEY_PATH`, `GH_TOKEN`, `GIT_AUTHOR_NAME`, `GIT_COMMITTER_NAME`.

`KEY_PATH` is resolved against `$SCRIPT_DIR` if relative (cross-repo cwd safety per macf#140 + macf#161 — without this, `cd`-ing to another repo breaks the token helper and triggers the attribution trap).

`GH_TOKEN` is minted via the fail-loud `macf-gh-token.sh` helper (validates the `ghs_` prefix, surfaces diagnostics for clock drift / bad key / wrong App or install ID). Inlining the bare CLI here would re-create the attribution trap (no `pipefail`, failed fetch piped through `jq` succeeds, `GH_TOKEN` becomes the string `"null"`, `gh` falls back to stored user auth) — see `coordination.md` Token & Git Hygiene.

**Local-mode placeholder** (per [DR-024](../design/decisions/DR-024-local-registry-mode.md) / macf#322): in local-registry mode, this file's body is just a comment explaining why no exports are emitted. Local mode does not mint a GitHub App installation token — coordination uses the local registry file at `$MACF_REGISTRY_PATH`, and commits land as the local user (not as `app/<bot>[bot]`).

**Common operator edits**: do not edit. To rotate App credentials, update `.macf/macf-agent.json` and re-run `macf update`. To migrate from GitHub mode to local mode (or vice versa), re-run `macf init` with the appropriate `--registry-type` flag — see [DR-024 §"Migration path"](../design/decisions/DR-024-local-registry-mode.md).

### `env.certs` — macf-managed

**Owner**: macf-managed.

**Vars defined**: `MACF_CA_CERT`, `MACF_CA_KEY`, `MACF_AGENT_CERT`, `MACF_AGENT_KEY`, `MACF_LOG_PATH`.

CA paths differ by registry mode: local-mode CAs live next to the registry file (`~/.macf/registry/<project>.ca.{crt,key}`); GitHub-mode CAs live under `~/.macf/certs/<project>/`. Both modes need `MACF_CA_CERT` / `MACF_CA_KEY` exported for the channel server's mTLS bootstrap.

**Common operator edits**: do not edit. To rotate certs, use `macf certs rotate`. To recover from CA-key loss, see `macf certs recover`.

**Example content** (GitHub-mode):

```bash
export MACF_CA_CERT="$HOME/.macf/certs/my-project/ca-cert.pem"
export MACF_CA_KEY="$HOME/.macf/certs/my-project/ca-key.pem"
export MACF_AGENT_CERT="$SCRIPT_DIR/.macf/certs/agent-cert.pem"
export MACF_AGENT_KEY="$SCRIPT_DIR/.macf/certs/agent-key.pem"
export MACF_LOG_PATH="$SCRIPT_DIR/.macf/logs/channel.log"
```

### `env.registry` — macf-managed

**Owner**: macf-managed.

**Vars defined**: `MACF_REGISTRY_TYPE` + one of `MACF_REGISTRY_REPO` / `MACF_REGISTRY_ORG` / `MACF_REGISTRY_USER` / `MACF_REGISTRY_PATH` depending on type.

The plugin's `src/config.ts` reads these vars on startup. Without them the plugin falls back to a hardcoded default repo and 403s every registry op for consumers in other scopes (see macf#178).

**Common operator edits**: do not edit. To change registry scope (e.g., move from `--registry-type repo` to `--registry-type org`), re-run `macf init` with the new flag.

**Example content** (repo-scope):

```bash
export MACF_REGISTRY_TYPE="repo"
export MACF_REGISTRY_REPO="my-org/coordination-repo"
```

### `env.telemetry` — operator-managed

**Owner**: operator-managed. Bootstrap-written on first `macf update` if absent; preserved unconditionally otherwise.

**Vars defined**: `CLAUDE_CODE_ENABLE_TELEMETRY`, `CLAUDE_CODE_ENHANCED_TELEMETRY_BETA`, `OTEL_TRACES_EXPORTER`, `OTEL_METRICS_EXPORTER`, `OTEL_LOGS_EXPORTER`, `MACF_OTEL_ENDPOINT`, `OTEL_EXPORTER_OTLP_ENDPOINT`, `OTEL_EXPORTER_OTLP_PROTOCOL`, `OTEL_SERVICE_NAME`, `OTEL_RESOURCE_ATTRIBUTES`.

Three independent telemetry-signal gates — ALL required for the corresponding signal to emit:

- `CLAUDE_CODE_ENABLE_TELEMETRY` — master telemetry gate
- `CLAUDE_CODE_ENHANCED_TELEMETRY_BETA` — additional gate for traces
- `OTEL_TRACES_EXPORTER=otlp` / `OTEL_METRICS_EXPORTER=otlp` / `OTEL_LOGS_EXPORTER=otlp` — per-signal exporter (without these, the signal silently emits nothing even when the master gate is on; macf#245 surfaced the metrics+logs gap)

`MACF_OTEL_ENDPOINT` defaults to `http://localhost:14318` (canonical k3d-cluster endpoint per [`groundnuty/macf-devops-toolkit`](https://github.com/groundnuty/macf-devops-toolkit)). The 4-layer endpoint resolution chain (env > settings.local.json > template-time bake > hardcoded default) lives in this file's `MACF_OTEL_ENDPOINT="${MACF_OTEL_ENDPOINT:-...}"` lines (macf#313).

**Minimal placeholder** when `MACF_OTEL_DISABLED=1` is set at `macf init` / `macf update` time (per macf#197): the body shrinks to a comment explaining the opt-out — no OTel exports emitted, sourcing the file is a no-op.

**Common operator edits**:

- **Customize the OTLP endpoint**: edit `MACF_OTEL_ENDPOINT` (e.g., `export MACF_OTEL_ENDPOINT="https://your-collector.example.com/v1/traces"`). Survives `macf update`.
- **Turn off a single signal**: comment out the relevant `OTEL_<SIGNAL>_EXPORTER=otlp` line.
- **Change service name / resource attrs**: edit `OTEL_SERVICE_NAME` or `OTEL_RESOURCE_ATTRIBUTES`. The defaults bake `gen_ai.agent.name=<agent>,gen_ai.agent.role=<role>,service.namespace=macf` for cross-agent dashboarding.

### `env.tmux` — operator-managed

**Owner**: operator-managed. Bootstrap-written on first `macf update` if absent; preserved unconditionally otherwise.

**Vars defined**: `MACF_TMUX_SESSION`, `MACF_TMUX_WINDOW`.

Used by the on-notify wake path (macf#185): when the channel server's `onNotify` handler fires, it shells out to `tmux-send-to-claude.sh <session>:<window> <prompt>` to inject the notification as the TUI's next input turn.

**Minimal placeholder** when neither `tmux_session` nor `tmux_window` is set in `.macf/macf-agent.json`: the wake path falls back to auto-detect from `$TMUX` (existing behavior — no env var needed when the channel server runs inside a tmux pane).

**Common operator edits**:

- **Target a specific named pane** (e.g., agent launched outside tmux by a supervisor): set `MACF_TMUX_SESSION` and `MACF_TMUX_WINDOW`.

**Example content**:

```bash
export MACF_TMUX_SESSION="my-project"
export MACF_TMUX_WINDOW="code-agent"
```

## Common operator workflows

### Customize the OTLP endpoint

Edit `.claude/.macf/env.telemetry`:

```bash
export MACF_OTEL_ENDPOINT="https://your-collector.example.com/v1/traces"
```

Restart `claude.sh` — the new endpoint takes effect on next launch. `env.telemetry` is operator-managed, so `macf update` preserves your edit unconditionally.

### Customize tmux session/window targets

Edit `.claude/.macf/env.tmux`:

```bash
export MACF_TMUX_SESSION="custom-session"
export MACF_TMUX_WINDOW="custom-window"
```

Same persistence semantics as `env.telemetry`.

### Override `MACF_AGENT_NAME` for an ad-hoc run

Three options, by persistence:

1. **Per-invocation**: `MACF_AGENT_NAME=foo ./claude.sh` — env var wins because of the 3-layer priority in `env.identity`.
2. **Persistent via settings**: add `env.MACF_AGENT_NAME: "foo"` to `.claude/settings.local.json`. Read at runtime by `macf_settings_get`; survives `macf update` of `env.identity` (which would otherwise rewrite the baked default). This was the canonical pre-#342 override surface; post-#342 it still works as a backward-compat shim, but emits a one-time deprecation warning during `macf update` (see [settings.local.json env block](#settingslocaljson-env-block-option-α-clean-break-with-deprecation-shim) below).
3. **Persistent via local file**: add `.claude/.macf/env.local.identity` with `export MACF_AGENT_NAME=foo`. Sources after canonical `env.identity` per the post-canonical sort order (see [Operator-custom convention](#operator-custom-convention) below).

### Add custom env vars

Create `.claude/.macf/env.local.<name>` with your `export` lines. The thin claude.sh source-loop picks them up automatically — no claude.sh edit needed.

Example: a per-host telemetry override committed only on a specific operator's machine:

```bash
# .claude/.macf/env.local.my_overrides
export MACF_OTEL_ENDPOINT="https://custom.observability.example.com/"
export OTEL_RESOURCE_ATTRIBUTES="${OTEL_RESOURCE_ATTRIBUTES},host.role=development"
```

## Operator-custom convention

Operator-added env files must use the `env.local.<name>` or `env.zz.<name>` prefix to ensure they sort AFTER all macf-managed canonical files in the bash glob expansion.

**Why prefix matters**: bash glob ASCII sort puts uppercase A-Z (0x41-0x5A) BEFORE underscore (0x5F) BEFORE lowercase a-z (0x61-0x7A). So an operator-added `env.UPPERCASE_OVERRIDE` would sort BEFORE `env._helpers` and source in a context without `macf_settings_get` defined yet — followed by `env.identity` calling that undefined function. Stick to `env.local.*` or `env.zz.*` to avoid the trap.

Sort order for the canonical seven files plus operator extensions:

```
env._helpers       ← sourced first (underscore < letters)
env.certs
env.github
env.identity
env.registry
env.telemetry
env.tmux
env.local.<name>   ← sourced AFTER canonical files (lowercase 'l' > '_')
env.zz.<name>      ← sourced last
```

Later files can override variables exported by earlier ones — bash `export` is idempotent on re-assignment.

## Rollback

If migration to the multi-file layout caused issues in your workspace, you can roll back to the pre-#342 monolithic `claude.sh`:

```bash
# Restore monolithic claude.sh from git
git checkout HEAD~1 -- claude.sh

# Remove generated env files
rm -rf .claude/.macf/env.*
```

The next `macf update` (without `--no-migrate-env-files`) will re-attempt the migration. To skip migration on a specific run, pass `--no-migrate-env-files`:

```bash
macf update --no-migrate-env-files
```

You'll miss launcher-template evolution that run; remove the flag on the next `macf update` to re-attempt the migration.

**State equivalence**:

- Monolithic `claude.sh` + no `.claude/.macf/env.*` files = working v0.2.17-style workspace
- Thin `claude.sh` + `.claude/.macf/env.*` files = v0.2.18+ workspace

Both shapes are functional; the architectural difference is operator UX (per-concern editing, refresh boundaries) — not behavior. The auto-migration is opt-out (`--no-migrate-env-files`) precisely so workspaces with established v0.2.17 state can defer the upgrade if needed.

## settings.local.json env block — Option α (clean break with deprecation shim)

**History**: pre-#342, the canonical operator override surface for `MACF_*` and `OTEL_*` env vars was the `.claude/settings.local.json` `env` block, read at runtime by `macf_settings_get`.

**Post-#342**: the per-concern env files are the canonical surface. The `env` block in `settings.local.json` is structurally still read at runtime (backward-compat shim — `macf_settings_get` continues to query the JSON), but `macf update` emits a one-time deprecation warning listing each `env.MACF_*` or `env.OTEL_*` key it found:

```
Warning: deprecated env key(s) in .claude/settings.local.json (macf#342):
  env.MACF_AGENT_NAME
  env.MACF_OTEL_ENDPOINT
  These are still read at runtime by macf_settings_get (backward-compat),
  but the canonical location is now .claude/.macf/env.* (or env.local.*
  for operator overrides). See macf#342.
```

**Migration**: operator manually moves the keys from JSON to the matching `env.*` file. For example, `env.MACF_OTEL_ENDPOINT` in JSON → edit the `MACF_OTEL_ENDPOINT` line in `env.telemetry` (or add to `env.local.telemetry`).

**No automatic JSON-key migration in v0.2.18** (risk-conservative). Operators may have intentional layered configurations where the JSON value differs from the file default (e.g., dev endpoint in JSON, prod default baked into file). Auto-moving the value loses that distinction. Manual migration after seeing the warning is the conservative choice.

**Hard-deprecation timeline**: the warning-only shim continues until a future major version bump. At that point the `macf_settings_get` read of `env.MACF_*` / `env.OTEL_*` keys becomes a hard error rather than a soft warning. No specific timeline; the upgrade follows the operator-feedback signal — if substantial workspaces still depend on the JSON path after a release cycle or two, the deprecation window extends.
